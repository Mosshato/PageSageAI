MODULE 1: Authentication Module
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

Module Name:
  Authentication Module (auth_views.py + AuthContext.jsx + ProtectedRoute.jsx)

Purpose:
  Manages user identity across the entire platform. Responsible for registering
  new users, validating credentials, issuing JWT tokens, refreshing expired
  access tokens, and enforcing role-based access control on both the frontend
  and backend. Ensures that student and teacher routes are fully separated and
  that unauthenticated requests never reach protected resources.

Inputs:
  - POST /api/signup/       : { email, password, firstName, lastName, role }
  - POST /api/login/        : { email, password }
  - POST /api/token/refresh/: { refresh_token }
  - GET  /api/me/           : Authorization: Bearer <access_token>

Outputs:
  - On signup/login  : { access_token, refresh_token, role, firstName }
  - On /me           : { id, email, firstName, lastName, role }
  - On refresh       : { access_token }
  - On failure       : HTTP 400 / 401 with error message

Dependencies:
  Internal:
    - api/models.py            : User model (email-based, role field)
    - core/settings.py         : SIMPLE_JWT config (access=5min, refresh=1day)
  External:
    - djangorestframework-simplejwt : token generation and validation
    - React AuthContext.jsx         : stores tokens in localStorage, exposes
                                      user state globally
    - ProtectedRoute.jsx            : redirects unauthenticated users to /login

Flow:
  [User fills Login form]
          |
          v
  POST /api/login/
          |
    ┌─────┴─────┐
  valid?        no
    |            └──> HTTP 401 "Invalid credentials"
    v
  Issue access_token (5 min) + refresh_token (1 day)
          |
          v
  Frontend stores tokens in localStorage
          |
          v
  AuthContext reads role from token payload
          |
     ┌────┴────┐
  student    teacher
     |            |
  /student/*   /teacher/*
     |            |
  ProtectedRoute checks role on every navigation
          |
          v
  On 401 response → auto-call /api/token/refresh/
          |
    ┌─────┴─────┐
  success      fail
    |            └──> clear localStorage → redirect /login
    v
  Retry original request with new access_token

MODULE 2: Class & Enrollment Module
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

Module Name:
  Class & Enrollment Module (views.py — class/enrollment endpoints)

Purpose:
  Allows teachers to create and manage classes, and students to discover and
  join classes using a unique enrollment code. Acts as the organizational
  backbone of the platform — every assignment, lecture, announcement, and
  AI course is scoped to a class.

Inputs:
  Teacher:
    - POST /api/teacher/classes/            : { name, description }
    - DELETE /api/teacher/classes/<id>/     : (teacher must own class)
  Student:
    - GET  /api/classes/                    : (returns enrolled classes)
    - POST /api/classes/enroll/             : { code }
    - DELETE /api/classes/<id>/unenroll/    : (student leaves class)

Outputs:
  - Class list for teacher/student
  - Created class object with generated enrollment code
  - Enrollment confirmation or error

Dependencies:
  Internal:
    - api/models.py     : Class, Enrollment models
    - api/serializers.py: ClassSerializer (teacher vs student variants)
    - JWT middleware     : request.user ownership checks

Flow:
  [Teacher creates class]
          |
          v
  POST /api/teacher/classes/ { name, description }
          |
          v
  System generates unique 6-char enrollment code (e.g. DEMO01)
          |
          v
  Class saved to PostgreSQL
          |
          v
  Teacher shares code with students
          |
          v
  [Student enters code]
          |
          v
  POST /api/classes/enroll/ { code }
          |
    ┌─────┴─────┐
  valid?        no
    |            └──> HTTP 404 "Class not found"
    v
  Enrollment record created (student ↔ class)
          |
          v
  Class appears in student's dashboard

Module Name:
  Assignment & Submission Module (views.py — assignment/submission endpoints)

Purpose:
  Manages the full lifecycle of academic assignments: creation by teachers,
  file attachment, student submission, resubmission, unsubmit, and grading.
  Tracks a denormalized status per assignment reflecting the most recent
  student action.

Inputs:
  Teacher:
    - POST   /api/teacher/classes/<id>/assignments/  : { title, description, due_date }
    - POST   /api/teacher/assignments/<id>/attachments/ : file upload
    - POST   /api/teacher/submissions/<id>/grade/    : { grade, feedback }
  Student:
    - GET    /api/assignments/                       : list all assignments
    - POST   /api/assignments/<id>/submit/           : file upload(s)
    - POST   /api/assignments/<id>/unsubmit/         : (removes submission)

Outputs:
  - Assignment objects with status: pending | submitted | graded
  - Submission objects with uploaded files
  - Grade and feedback after teacher review

Dependencies:
  Internal:
    - api/models.py       : Assignment, AssignmentAttachment, Submission,
                            SubmissionFile models
    - api/serializers.py  : AssignmentSerializer, SubmissionSerializer
    - media/submissions/  : file storage for student uploads
    - media/attachments/  : file storage for teacher attachments
  External:
    - Django file handling (InMemoryUploadedFile)
    - AssignmentsContext.jsx : frontend cache to avoid refetching

Flow:
  [Teacher creates assignment]
          |
          v
  POST /api/teacher/classes/<id>/assignments/
          |
          v
  Assignment saved, status = "pending"
          |
          v
  Students see assignment in their dashboard
          |
          v
  [Student submits files]
          |
          v
  POST /api/assignments/<id>/submit/
          |
          v
  SubmissionFile records created → media/submissions/
  Assignment status → "submitted"
          |
          v
  [Teacher reviews submission]
          |
          v
  POST /api/teacher/submissions/<id>/grade/ { grade, feedback }
          |
          v
  Assignment status → "graded"
          |
          v
  [Student can resubmit → status resets to "submitted"]

MODULE 5: PDF Ingestion Module
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

Module Name:
  PDF Ingestion Module (textAndImageExtraction.py — ingest_pdf)

Purpose:
  Transforms a raw PDF file into a structured, per-page representation
  suitable for downstream narration and RAG indexing. Renders each page
  as a high-resolution PNG, extracts text blocks with spatial coordinates,
  and classifies each block semantically using a local transformer model.

Inputs:
  - pdf_path  : absolute path to the uploaded PDF file
  - output_dir: target directory for output files

Outputs:
  - page_0001.png ... page_XXXX.png    : rendered page images (200 DPI)
  - page_0001.json ... page_XXXX.json  : structured block data per page
    Each JSON contains: { page_number, width, height, blocks: [
      { text, type, bbox, font_size, is_bold }
    ]}
  - course_index.json : { total_pages, pages: [ { page_number, png, json } ] }

Dependencies:
  Internal:
    - output_dir structure (created by pipeline)
  External:
    - PyMuPDF (fitz)                      : PDF rendering and text extraction
    - Pillow (PIL)                         : image conversion to RGB
    - transformers (HuggingFace)           : LayoutLMv3Processor,
                                            LayoutLMv3ForTokenClassification
    - torch                                : model inference (GPU if available)
    - microsoft/layoutlmv3-base            : semantic block classification model
                                            (~900MB, auto-downloaded on first run)

Flow:
  [Receive pdf_path, output_dir]
          |
          v
  Load LayoutLMv3 model (once, cached in memory)
          |
          v
  FOR each page in PDF:
    |
    ├─> Render page → RGB image (200 DPI) → save page_XXXX.png
    |
    ├─> Extract text spans with (x0, y0, x1, y1, text, font_size, bold)
    |
    ├─> Normalize bounding boxes to 0-1000 scale (LayoutLMv3 convention)
    |
    ├─> Run LayoutLMv3 inference on (image, words, bboxes)
    |       token limit: 512 — truncate if page is very dense
    |
    ├─> Majority vote per text block → semantic type:
    |       title | paragraph | formula | figure | table | bullet | other
    |
    ├─> Fallback heuristics for ambiguous blocks
    |       (font size, bold flag, position on page)
    |
    └─> Save page_XXXX.json with classified blocks
          |
          v
  Save course_index.json with total_pages and file manifest
          |
          v
  Return to ai_pipeline.py (Step 1 complete)

MODULE 6: Narration Generation Module
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

Module Name:
  Narration Generation Module (tableExtraction.py — narrate_course)

Purpose:
  Converts the structured JSON blocks extracted from each PDF page into
  natural-language narration text, as if spoken by a university professor.
  Processes pages sequentially, skipping those already narrated (idempotent
  resume support). Each narration is self-contained and tuned to the
  estimated reading/speaking duration of the slide.

Inputs:
  - output_dir : path to directory containing page_XXXX.json files
  - model      : LLM model ID (default: "llama-3.1-8b-instant")
  - api_key    : Groq API key
  - seconds_per_slide : target narration duration (default: 120s)

Outputs:
  - page_0001_narration.json ... page_XXXX_narration.json
    Each file contains: { page_number, narration_text, word_count,
                          estimated_duration_seconds }

Dependencies:
  Internal:
    - page_XXXX.json files (output of PDF Ingestion Module)
  External:
    - groq (Python SDK)          : API client for Groq inference
    - llama-3.1-8b-instant       : LLM hosted on Groq (fast inference)
    - python-dotenv              : optional key loading from .env

Flow:
  [Receive output_dir]
          |
          v
  Scan output_dir for all page_XXXX.json files
          |
          v
  FOR each page JSON (sorted by page number):
    |
    ├─> Check if page_XXXX_narration.json already exists
    |       YES → skip (idempotent resume)
    |       NO  → continue
    |
    ├─> Read classified blocks from page_XXXX.json
    |
    ├─> Build prompt:
    |     "You are a university professor. Given these slide elements,
    |      generate a spoken narration of ~{seconds}s. Be clear,
    |      educational, and complete."
    |
    ├─> Call Groq API (llama-3.1-8b-instant)
    |       Retry on RateLimitError (max 3 attempts, exponential backoff)
    |
    └─> Save page_XXXX_narration.json
          |
          v
  All pages narrated → return to ai_pipeline.py (Step 2 complete)

MODULE 7: Text-to-Speech Module
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

Module Name:
  Text-to-Speech Module (TTS.py — generate_audio_for_course)

Purpose:
  Converts narration text for each course page into MP3 audio files using
  the Deepgram Aura-2 TTS API. Produces spoken audio at 24kHz that students
  listen to while browsing the AI-powered lecture viewer. Supports idempotent
  resume and exponential backoff on API failures.

Inputs:
  - output_dir   : path containing page_XXXX_narration.json files
  - voice        : Deepgram voice ID (default: "aura-2-orpheus-en")
  - api_key      : Deepgram API key

Outputs:
  - output_dir/audio/page_0001.mp3 ... page_XXXX.mp3
    Audio files at 24kHz, mp3 encoding, one per page

Dependencies:
  Internal:
    - page_XXXX_narration.json (output of Narration Module)
    - media/ai_courses/<id>/audio/ directory
  External:
    - requests               : HTTP client for Deepgram REST API
    - Deepgram API           : https://api.deepgram.com/v1/speak
    - Deepgram Aura-2 model  : cloud TTS engine

Flow:
  [Receive output_dir, voice, api_key]
          |
          v
  Create audio/ subdirectory if not exists
          |
          v
  FOR each page_XXXX_narration.json (sorted):
    |
    ├─> Check if audio/page_XXXX.mp3 exists AND size > 0
    |       YES → skip (idempotent resume)
    |       NO  → continue
    |
    ├─> Read narration_text from JSON
    |
    ├─> POST https://api.deepgram.com/v1/speak
    |     Headers: Authorization: Token <api_key>
    |     Params:  model=<voice>, encoding=mp3, sample_rate=24000
    |     Body:    { text: narration_text }
    |
    ├─> On success: stream response bytes → write page_XXXX.mp3
    |
    └─> On failure: retry up to 3 times with exponential backoff
          |
          v
  All MP3s generated → return to ai_pipeline.py (Step 3 complete)

MODULE 8: RAG Indexing & Query Module
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

Module Name:
  RAG Indexing & Query Module (ai_pipeline.py — build_course_chroma,
                                query_rag, reformulate)

Purpose:
  Indexes the narration text of each course page into a ChromaDB vector
  database, enabling semantic search. When a student asks a question,
  retrieves the most relevant chunks and passes them to Gemini for a
  contextual answer. Also supports simplified re-explanation of any page.

Inputs:
  Indexing:
    - output_dir   : path with page_XXXX_narration.json files
    - course_id    : integer, used as chunk prefix
  Querying:
    - POST /api/ai-courses/<id>/ask/          : { question }
    - POST /api/ai-courses/<id>/reformulate/  : { page_number }

Outputs:
  - ChromaDB populated with course chunks (marker: chroma_indexed.marker)
  - ask/        : { answer: "..." }
  - reformulate/: { simplified_text: "..." }

Dependencies:
  Internal:
    - page_XXXX_narration.json files
    - Functionalities/RAG/chroma_ossu_db/  : shared ChromaDB instance
                                             (must be seeded first via createDB.py)
  External:
    - chromadb                       : local vector database client
    - langchain-chroma               : LangChain ↔ ChromaDB integration
    - langchain-huggingface          : HuggingFaceEmbeddings wrapper
    - all-MiniLM-L6-v2              : local embedding model (80MB)
    - langchain-google-genai         : ChatGoogleGenerativeAI wrapper
    - gemini-2.5-flash               : LLM for answer synthesis
    - langchain-core                 : PromptTemplate, RunnablePassthrough,
                                       StrOutputParser, RunnableParallel
    - langchain-text-splitters       : RecursiveCharacterTextSplitter
                                       (chunk_size=700, overlap=100)

Flow — Indexing (Step 4 of AICourse pipeline):
  [Receive output_dir, course_id]
          |
          v
  Check chroma_indexed.marker → if exists, skip
          |
          v
  Load all page_XXXX_narration.json → extract narration_text
          |
          v
  Split into chunks (700 chars, 100 overlap)
  Prefix each chunk: "[Course {id} | Page {n}] ..."
          |
          v
  Generate embeddings with all-MiniLM-L6-v2
          |
          v
  Add chunks to shared ChromaDB (chroma_ossu_db/)
          |
          v
  Write chroma_indexed.marker → Step 4 complete

Flow — Query (student asks a question):
  [Student sends POST /ask/ { question }]
          |
          v
  Generate embedding for question (all-MiniLM-L6-v2)
          |
          v
  ChromaDB similarity search → top-k relevant chunks
          |
          v
  Build prompt: context chunks + question
          |
          v
  Call Gemini gemini-2.5-flash
          |
          v
  Return { answer } to frontend

MODULE 9: AICourse Pipeline Orchestrator
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

Module Name:
  AICourse Pipeline Orchestrator (ai_pipeline.py — run_ai_pipeline,
                                   apps.py — ApiConfig.ready)

Purpose:
  Orchestrates the four AI processing steps (PDF ingestion, narration, TTS,
  ChromaDB indexing) as a single resumable pipeline running in a background
  daemon thread. Persists pipeline state in the AICourse database record so
  the frontend can poll progress and crashed pipelines can be resumed on
  server restart.

Inputs:
  - Teacher POST /api/teacher/classes/<id>/ai-courses/  : PDF file upload
  - AICourse DB record with: id, output_dir, status, current_step

Outputs:
  - AICourse.status   : PROCESSING → READY | ERROR
  - AICourse.current_step: PENDING → EXTRACTING → NARRATING → TTS → CHROMA → DONE
  - AICourse.total_pages  : page count (set after step 1)
  - Media files in media/ai_courses/<id>/

Dependencies:
  Internal:
    - Module 5: PDF Ingestion (ingest_pdf)
    - Module 6: Narration Generation (narrate_course)
    - Module 7: Text-to-Speech (generate_audio_for_course)
    - Module 8: RAG Indexing (build_course_chroma)
    - api/models.py : AICourse model
  External:
    - Python threading.Thread (daemon=True)
    - apps.py ApiConfig.ready() for startup resume

Flow:
  [Teacher uploads PDF]
          |
          v
  Create AICourse record (status=PROCESSING, step=PENDING)
  Save PDF to media/ai_courses/<id>/
  Start daemon thread: run_ai_pipeline(course_id)
          |
          v
  ┌──────────────────────────────────────┐
  │ STEP 1: EXTRACTING                   │
  │  Check idempotency → run ingest_pdf  │
  │  Update step=EXTRACTING in DB        │
  └──────────────────────────────────────┘
          |
          v
  ┌──────────────────────────────────────┐
  │ STEP 2: NARRATING                    │
  │  Check idempotency → run narrate_    │
  │  course (Groq)                       │
  │  Update step=NARRATING in DB         │
  └──────────────────────────────────────┘
          |
          v
  ┌──────────────────────────────────────┐
  │ STEP 3: TTS                          │
  │  Check idempotency → run generate_   │
  │  audio_for_course (Deepgram)         │
  │  Update step=TTS in DB               │
  └──────────────────────────────────────┘
          |
          v
  ┌──────────────────────────────────────┐
  │ STEP 4: CHROMA                       │
  │  Check marker → run build_course_    │
  │  chroma (ChromaDB)                   │
  │  Update step=CHROMA in DB            │
  └──────────────────────────────────────┘
          |
     ┌────┴────┐
   success    error
     |            └──> AICourse.status = ERROR
     v               AICourse.error_msg = traceback
  AICourse.status = READY
  AICourse.current_step = DONE
          |
          v
  Frontend polling /status/ detects READY → unlocks AI Teach tab

  On server restart (apps.py):
    Query all AICourse where status=PROCESSING
    → restart daemon thread for each

MODULE 10: Concept Animation Module
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

Module Name:
  Concept Animation Module (manim_pipeline.py)

Purpose:
  Generates short educational Manim animations on-demand when a student
  requests a visual explanation of a concept. Gemini produces the Python
  Manim code; the Manim CLI renders it into an MP4 video. Results are
  cached by concept key so repeated requests for the same concept are
  served instantly.

Inputs:
  - POST /api/ai-courses/<id>/animations/  : { concept: "Fourier Transform" }
  - ConceptAnimation DB record (id, concept, concept_key, ai_course)

Outputs:
  - ConceptAnimation.status   : PENDING → GENERATING → RENDERING → READY | ERROR
  - ConceptAnimation.video_file: relative path to .mp4 in media/
  - GET /animations/<id>/status/ returns { video_url } when READY

Dependencies:
  Internal:
    - api/models.py : ConceptAnimation model
    - media/animations/anim_<id>/ : working directory per animation
  External:
    - google-genai SDK          : Gemini gemini-2.5-flash for code generation
    - Manim Community CLI       : subprocess render (must be on PATH)
    - Python subprocess         : manim -ql --media_dir scene.py ConceptScene
    - threading.Thread          : daemon thread per animation

Flow:
  [Student clicks Animate on a concept]
          |
          v
  POST /api/ai-courses/<id>/animations/ { concept }
          |
          v
  concept_key = concept.strip().lower()
  Check DB: existing READY animation for same (ai_course, concept_key)?
     YES → return cached video_url immediately
     NO  → create ConceptAnimation record, start daemon thread
          |
          v
  ┌─────────────────────────────────────┐
  │ STEP 1: GENERATING                  │
  │  Build prompt for Gemini:           │
  │  "Generate Manim Python code for    │
  │   ConceptScene class that visually  │
  │   explains: {concept}"              │
  │  Call gemini-2.5-flash              │
  │  Strip markdown code fences         │
  │  Write scene.py to anim_<id>/       │
  └─────────────────────────────────────┘
          |
          v
  ┌─────────────────────────────────────┐
  │ STEP 2: RENDERING                   │
  │  subprocess: manim -ql              │
  │    --media_dir anim_<id>/           │
  │    scene.py ConceptScene            │
  │  Timeout: 180 seconds               │
  └─────────────────────────────────────┘
          |
     ┌────┴────┐
   success    error / timeout
     |            └──> retry_count += 1
     v               if retry_count < 3 → restart from GENERATING
  Find .mp4 in media_dir               else → status = ERROR
  Save relative path to DB
  status = READY
          |
          v
  Frontend polling every 4s detects READY
  Switches to Animations tab, plays video

MODULE 11: Quiz Module
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

Module Name:
  Quiz Module (quiz_pipeline.py + views.py — quiz endpoints)

Purpose:
  Automatically generates a multiple-choice quiz from the narration content
  of an AICourse. Teachers trigger generation; students take the quiz once
  and receive an instant score. Teachers can view per-student results.

Inputs:
  Teacher:
    - POST /api/teacher/classes/<id>/ai-courses/<id>/generate-quiz/
    - GET  /api/teacher/classes/<id>/ai-courses/<id>/quiz-results/
  Student:
    - GET  /api/ai-courses/<id>/quiz/status/
    - GET  /api/ai-courses/<id>/quiz/questions/
    - POST /api/ai-courses/<id>/quiz/attempt/  : { answers: [0,2,1,3,...] }

Outputs:
  - Quiz.questions : JSON array of 20 objects:
      { question, options[4], correct_index }
  - QuizAttempt   : { score (%), correct_count, total }
  - Teacher results: list of { student_name, score, correct_count, date }

Dependencies:
  Internal:
    - api/models.py : Quiz (OneToOneField → AICourse), QuizAttempt models
    - page_XXXX_narration.json files (max 40,000 chars total)
  External:
    - google-genai SDK      : Gemini gemini-2.5-flash
    - threading.Thread      : daemon thread for generation
    - Python json           : parse and validate Gemini response

Flow — Generation:
  [Teacher clicks Generate Quiz]
          |
          v
  POST /api/teacher/.../generate-quiz/
          |
          v
  Create Quiz record (status=PENDING)
  Start daemon thread: run_quiz_pipeline(quiz_id)
          |
          v
  Collect all page_XXXX_narration.json → concatenate text
  Truncate to 40,000 chars if needed
          |
          v
  Build prompt:
    "Generate exactly 20 multiple-choice questions from this content.
     Return a JSON array: [{question, options[4], correct_index}, ...]"
          |
          v
  Call Gemini gemini-2.5-flash
          |
          v
  Parse JSON response
  Validate: 20 items, each has 4 options, correct_index in 0-3
          |
     ┌────┴────┐
   valid      invalid
     |            └──> Quiz.status = ERROR
     v
  Quiz.questions = validated array
  Quiz.status = READY

Flow — Student attempt:
  [Student submits answers]
          |
          v
  POST /api/ai-courses/<id>/quiz/attempt/ { answers: [...] }
          |
          v
  Check: student already attempted? → HTTP 400
          |
          v
  Compare answers[i] vs questions[i].correct_index for each of 20
  Compute correct_count, score = correct_count / 20 * 100
          |
          v
  Save QuizAttempt record
  Return { score, correct_count, total: 20 }

━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

Module Name:
  ChromaDB Seeding Module (Functionalities/RAG/createDB.py)

Purpose:
  One-time setup script that populates the shared ChromaDB vector database
  with the OSSU (Open Source Society University) Computer Science curriculum.
  This baseline knowledge allows the RAG system to answer general CS questions
  even before any course PDF is uploaded.

Inputs:
  - OSSU README.md fetched from GitHub at runtime:
    https://raw.githubusercontent.com/ossu/computer-science/master/README.md

Outputs:
  - chroma_ossu_db/ : persistent ChromaDB directory containing OSSU chunks
    as vector embeddings

Dependencies:
  External:
    - requests                    : download OSSU README from GitHub
    - langchain-text-splitters    : RecursiveCharacterTextSplitter
                                    (chunk_size=700, overlap=100)
    - langchain-huggingface       : HuggingFaceEmbeddings wrapper
    - all-MiniLM-L6-v2           : local embedding model
    - langchain-chroma            : Chroma vector store integration
    - chromadb                    : persistence layer

Flow:
  [Run once before starting Django server]
          |
          v
  GET https://raw.githubusercontent.com/ossu/computer-science/master/README.md
          |
          v
  Split content into chunks (700 chars, 100 overlap)
          |
          v
  Load all-MiniLM-L6-v2 embedding model (local)
          |
          v
  Generate embeddings for all chunks
          |
          v
  Persist to chroma_ossu_db/ via ChromaDB
          |
          v
  DB ready — Django RAG queries can now use it
